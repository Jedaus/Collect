### ❓ 사용자 질문에 대한 답변: "웹사이트 만든 거 어디서 확인해요?"

**[기술적 설명]**
현재까지의 산출물(Mockup, 와이어프레임)은 **'최종 사용자에게 보여질 모습(Frontend/UI)'**의 완벽한 설계도입니다. 하지만 이 설계도에 생명력을 불어넣어 실제로 움직이는 웹사이트를 만들기 위해서는, 이 모든 데이터를 공급하는 **'신경망(Backend API)'** 구축이 선행되어야 합니다.

쉽게 말해, 디자인은 예쁜 옷이지만, 이 옷에 생명을 불어넣고 움직이게 하는 것은 **'데이터 엔진'**입니다. 현재 저희는 이 엔진의 핵심인 데이터 파이프라인 구축에 집중하고 있습니다.

### 🚧 1. 현재 개발 병목 지점 (The Bottleneck)

*   **문제:** 프론트엔드(Frontend, 웹사이트의 사용자 화면)는 Designer님과 Secretary님의 스펙에 따라 90%의 디자인과 구조가 확정되었습니다.
*   **해결 과제:** 이 모든 구조에 필요한 **실시간, 복합적인 데이터**를 가져올 API 엔드포인트가 부재합니다.
*   **핵심:** 아무리 아름다운 웹사이트라도, 데이터가 없으면 빈 껍데기입니다. 따라서 **데이터 파이프라인(Data Pipeline) 및 API 구축**이 현재 가장 높은 우선순위를 가집니다.

### 🚀 2. 핵심 기술 로드맵: '관계 그래프 분석 API' 완성 목표

저희는 가장 핵심적인 가치인 **'관계의 시간적 흐름 및 깊이'**를 처리할 전용 API를 최우선으로 개발할 것입니다.

| 단계 | 목표 기능 (API) | 기술적 역할 | 예상 완료 시점 |
| :--- | :--- | :--- | :--- |
| **Phase 1 (MVP)** | **관계 데이터 조회 API** (`/api/relationships/{id}`) | 특정 관계의 기본 정보(시작일, 마지막 상호작용 등)를 조회합니다. (가장 단순한 CRUD 기능) | 5월 5일 (다음 주 초) |
| **Phase 2 (핵심)** | **관계 그래프 분석 API** (`/api/relationship/graph`) | **[★최우선 목표]** 관계 테이블과 이벤트 테이블을 결합하여, 시간 경과에 따른 **관계의 강도 변화(Intensity Score)**를 계산하고 이를 JSON 형태로 반환합니다. | **5월 12일 (다음 주)** |
| **Phase 3 (고도화)** | **데이터 수집/정제 파이프라인** | 외부 입력(사용자 활동, 이벤트)을 받아 데이터베이스에 안정적으로 저장하고, 그래프 분석 API가 활용할 수 있도록 전처리합니다. | 5월 말 |
