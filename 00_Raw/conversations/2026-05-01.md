가제 - 콜렉

부제 - 다음은 너야 (기념일에 선물을 사서 리뷰를 올린 사람의 사진과 글을 보고 다음은 내 차례라는 기대감을 심어 준다)

목적: 매번 기념일 행사 때 원치 않는 선물을 받는 호스트와 어떤 선물을 줘야 할지 고민해야 하는 손님의 고민을 해결할 시스템.
손님들의 20$ ~ $30불 사이의 돈들을 쌓이는 플랫폼을 만들어서 호스트가 원하는 물건을 사게한다




기타 아이디어

카카오톡이나 페이스북 처럼 생일 알람 혹은 이벤트 알람을 울릴 수 있게 한다.
예를들어 사용자가 본인의 페이스북이나 카카오톡을 연동해 놓았고 그리고 사용자가 이벤트를 등록해놓으면 우리의 사용자가 본인의 친구들의 행사를 알게 되고 그것에 관련된 펀드라이징을 시작할 수 있게 한다. (친구들이 펀드라이징 링크를 보내는 것이 민망하다고 생각할 경우의 해결책)
혹은 이것을 문화화(Culturalizing) 해서 더 이상 선물로 돈을 보내는 것에 대한 거리낌을 해결할 수 있다.



추가적 아이디어들


아무나 이벤트를 등록할 수 있는 것이 아닌 약간의 절차과정을 만들어둔다
예를 들어 최대 한번 이상 기금을 했어야 이벤트를 등록할 수 있다. 혹은 카드 등록을 해야 이벤트를 등록할 수 있다. 


레벨 시스템을 마련해서 등급을 나눈뒤 각 등급별로 수수료 혜택을 차등한다.


이벤트 등록자가 콜렉 포인트로 선물을 사서 리뷰 형식으로 올린 글을 올리도록 유도한다. (올리면 포인트를 주는 방법 등등) 돈을 보내준 사람에게도 보람을 느낄 수 있게 모든 도너 한테 그 리뷰에 알람이 울리도록 한다.


나의 수익은 그들이 출금할때 그 금액의 5%를 차감하는 방식으로 한다.



일반 펀드라이즈와 콜렉의 차이점 

일반 펀드라이즈와 다른 것은 일반 펀드라이징은 불특정 다수에게 기금을 요구하는 것이지만
콜렉은 나의 지인들과 나의 친구들 (인스타,페이스북 친구들이)을 대상으로 기금을 요구하는 것이다.
즉 친구는 유대관계가 있기 때문에 기금을 할 가능성이 높다.
또한 이번에 내가 기금을 함으로써 다음 나의 생일에 받을 수 있을 거라는 기대감이 생긴다.





소비자 설득방법  


환경문제 해결, 친구들 선물 살 고민 해결, 생일자 본인도 원하는 물건을 얻을 수 있음
생일날에 친구들이 모아준 돈으로 본인이 원하는 선물(가격이 비싼)을 사게 되면 생일자가 더더욱 기억에 남는 생일이 될 것이다. 이 이벤트의 핵심은 생일자가 행복하게 하는 것이 주 목적인데 그것을 더욱 극대화 시켜줄 수 있기 때문이다. 또한 돈을 낸 친구들도 자신의 생일에 본인이 원하는 것을 받을 수 있는 기대감이 생긴다.

예를 들면 Teenager가 생일날 콜렉 플랫폼을 통해서 친구들에게 한사람당 대략$10씩 모으게 되었고 그 돈으로 본인이 원하는 것을 사게 된다 (아이폰, 에어팟 등등)
-> 과거 친구들로 부터 받은 쓸데 없는 물건들을 줄이고 본인이 원하는 고가의 물건 (평소에 사고 싶었던 것들을)을 원하는 금액 만큼은 못모을 수도 있지만 근접하게 모음으로서 살 수 있게 된다


문제점은 없을까?? 

아마 어떤 사람들은 선물은 정성인데 돈만 보내는 것은 성의 없다고 생각하여 꺼려 할 수도 있다. (모든사람을 다 포용할 순 없음) 콜렉의 문화가 자리 잡도록 주도한다
돈을 표시하는 방법이 문제가 될 수 있다. 만약 내 친구가 $30를 낼거라고 기대를 했는데 그것보다 아래의 돈을 냈다면 실망하게 될 수도 있고 친구들 관계가 멀어지게 될 수도 있다. 돈 문제는 약간 민감한 사항이므로 신중하게 문제 해결이 필요해 보임 (예를들면 금액 공개 여부의 관한 설정을 만들어서 이용자들에게 선택의 권한을 부여한다) 추가 문제 해결이 필요해 보임

### 💡 시장 적합성 (PMF) 평가
*   **평가:** **높음 (High Potential)**
*   **근거:** '선물 고민'과 '관계 유지'라는 인간의 근본적인 니즈를 건드림. 기존의 펀드라이징은 '도움'에 초점이 맞춰져 있어 감성적 결핍을 해결하지 못했지만, 콜렉은 '기대감(Expectation)'이라는 감성적 가치를 판매한다는 점에서 차별성이 확실함.
*   **위험 요소:** '금액 공개 민감성'과 '의무감 강요'로 인한 역효과가 발생할 수 있으므로, UX/UI 설계 시 **'선택권 부여'**가 최우선 과제임.

### 2. 초기 수익화 모델: 트랜잭션 피(Transaction Fee) 구조 설계

수익화는 단순한 거래 수수료가 아닌, **관계의 가치와 심리적 마찰(Psychological Friction)을 활용**하여 설계합니다. 이는 사용자들이 돈보다 '관계'에 더 집중하게 만드는 것을 목표로 합니다.

#### 2.1. 기본 수수료 구조 (Base Fee Structure)

모든 모금 활동에 대해 플랫폼이 가져가는 기본 수수료를 설정합니다.

*   **기본 플랫폼 수수료 (Platform Fee):** 총 모금액의 **5%** (이것은 플랫폼 운영 비용 및 관계 검증 비용으로 책정)
*   **거래 성공 수수료 (Transaction Success Fee):** 사용자가 성공적으로 기여를 완료했을 때 발생하는 수수료.

#### 2.2. 관계 기반 차등 수수료 로직 (Tiered Relationship Fee Logic)

수수료는 참여자와 기여의 **관계 깊이**에 따라 차등 적용됩니다. 이는 사용자가 '돈을 모으는 행위'를 넘어 '관계에 투자하는 행위'로 인식하게 만듭니다.

| 관계 등급 (Level) | 관계 기준 (예시) | 거래 성공 수수료율 | 심리적 효과 |
| :--- | :--- | :--- | :--- |
| **Level 1 (지인)** | 직접 초대 및 기여 (1:1 또는 소그룹) | **8%** | 친밀한 관계에 대한 보상으로 인식 |
| **Level 2 (그룹)** | 소규모 커뮤니티 내 기여 (5인 이내) | **6%** | 그룹 활동에 대한 협력의 가치 반영 |
| **Level 3 (커뮤니티)** | 공개된 캠페인 참여 및 아카이빙 기여 | **4%** | 플랫폼에 대한 신뢰 구축 및 충성도 확보 |

#### 2.3. 콜렉 포인트 지급 기준 (Collect Point Allocation)

수수료 외에, 관계 유지에 대한 보상으로 '콜렉 포인트'를 지급합니다.

*   **포인트 지급 기준:** 기여 금액의 **1%**를 콜렉 포인트로 전환하여 지급합니다. (예: 100만원 기여 시 10,000 포인트 지급)
*   **등급별 혜택:**
    *   **Level 1:** 포인트 적립 및 기본 추억 아카이빙 권한.
    *   **Level 2:** 우선적인 추억 공유 권한, 프리미엄 아카이빙 템플릿 접근.
    *   **Level 3:** 플랫폼 내에서 가장 높은 관계 등급 부여, 특별 이벤트 참여 기회 제공.

## 💾 2. 데이터베이스 스키마 설계 (PostgreSQL 중심)

가장 핵심적인 '관계 데이터'를 중심으로 테이블을 설계했습니다. 관계는 단순한 FK 연결이 아닌, **속성(Attributes)**을 가진 엔티티로 취급하여 '온도', '맥락', '심리적 가치' 등의 데이터를 기록합니다.

### 📌 핵심 테이블 구조 (Schema Diagram)

```sql
-- 1. User: 사용자 정보
CREATE TABLE User (
    user_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) UNIQUE NOT NULL,
    nickname VARCHAR(100) NOT NULL,
    profile_data JSONB, -- 생년월일, 지역 등 추가 정보 저장
    created_at TIMESTAMP DEFAULT NOW()
);

-- 2. Collection: 콜렉(행사/이벤트) 정보 (예: 생일, 결혼)
CREATE TABLE Collection (
    collection_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    host_id UUID REFERENCES User(user_id) ON DELETE CASCADE, -- 콜렉 주최자
    title VARCHAR(255) NOT NULL,
    event_date DATE NOT NULL,
    goal_amount DECIMAL(15, 2) DEFAULT 0.00, -- 모금 목표액
    description TEXT,
    status VARCHAR(50) DEFAULT 'ACTIVE' -- ACTIVE, COMPLETED, PAUSED
);

-- 3. Relationship: 관계 데이터 (가장 중요)
CREATE TABLE Relationship (
    relationship_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_user_id UUID REFERENCES User(user_id) ON DELETE CASCADE, -- 관계 시작점
    target_user_id UUID REFERENCES User(user_id) ON DELETE CASCADE, -- 관계 대상점
    collection_id UUID REFERENCES Collection(collection_id) ON DELETE CASCADE, -- 어떤 콜렉에서 발생했는지
    
    -- 관계의 속성 (Relationship Attributes)
    sentiment VARCHAR(50) NOT NULL, -- 'Warm', 'Formal', 'Deep' 등
    intensity DECIMAL(3, 2) NOT NULL, -- 0.00 ~ 1.00 (관계의 깊이)
    context_detail TEXT, -- 관계를 정의하는 구체적인 맥락 (예: "고등학교 시절 가장 친했던 친구")
    assigned_at TIMESTAMP DEFAULT NOW(),
    
    UNIQUE (source_user_id, target_user_id, collection_id) -- 같은 콜렉에서 중복 관계 방지
);

-- 4. Memory: 추억 아카이빙 (미디어 및 텍스트)
CREATE TABLE Memory (
    memory_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id UUID REFERENCES User(user_id) ON DELETE CASCADE, -- 메모 업로더
    collection_id UUID REFERENCES Collection(collection_id) ON DELETE CASCADE,
    
    media_type VARCHAR(50) NOT NULL, -- 'IMAGE', 'VIDEO', 'TEXT'
    media_url VARCHAR(512), -- AWS S3 등의 저장 경로
    caption TEXT, -- 사진/영상에 대한 캡션
    related_relationship_id UUID REFERENCES Relationship(relationship_id) ON DELETE SET NULL -- 이 메모가 어느 관계에 속하는지 연결
);

-- 5. Donation: 기여금 기록
CREATE TABLE Donation (
    donation_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    collection_id UUID REFERENCES Collection(collection_id) ON DELETE CASCADE,
    donor_user_id UUID REFERENCES User(user_id) ON DELETE RESTRICT,
    amount DECIMAL(15, 2) NOT NULL,
    donated_at TIMESTAMP DEFAULT NOW()
);
```
